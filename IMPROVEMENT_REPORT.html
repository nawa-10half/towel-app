2 コード品質 & アーキテクチャ

16件の改善点を検出。Singleton 依存によるテスト不能と、Swift 6 対応の不備が主な課題。

Critical Singleton Services でユニットテストが不可能 ▼

現状

全 7 サービスが static let shared + private init() のシングルトン。ViewModel や View から直接参照されており、モック差し替えができない。テストファイルは 0 件。

// FirestoreService.shared.towels を直接参照
FirestoreService.shared.deleteTowel(towelId)

推奨

Protocol-based Dependency Injection に移行。各サービスのプロトコルを定義し、ViewModel のイニシャライザ経由で注入する。デフォルト引数で .shared を指定すれば既存コードへの影響は最小限。

High サービス間の循環依存 ▼

現状

AuthService → GroupService → FirestoreService → GroupService の循環参照。変更時の影響範囲が予測困難。

推奨

AppCoordinator パターンでサービス間調整を一元管理するか、NotificationCenter / Combine で間接通信に移行。

High @unchecked Sendable の使用 — Swift 6 非準拠 ▼

現状

ConditionCheckService と NotificationService が @unchecked Sendable。Swift 6 の strict concurrency checking を無効化。

推奨

actor に変換するか、@MainActor 分離を明示的に適用。NotificationService は async/await 版 API (iOS 16+) に移行。

High エラーのサイレント無視が多数 ▼

現状

Alexa コード生成、表示名保存、交換記録の保存、通知の登録など、catch { } でエラーが完全に無視されている箇所が 5 箇所以上。ユーザーに操作失敗が伝わらない。

推奨

各所に errorMessage を設定しアラートまたはバナーで表示。特に ExchangeRecordSheet の保存失敗はデータロスに直結するため必須。

Medium MVVM パターンの不統一 ▼

ViewModel が存在するのは TowelListView と TowelDetailView のみ。他の画面 (TowelFormView, SettingsView, GroupSettingsView, SignInView 等) は View から直接 Service を呼んでおり、アーキテクチャが中途半端。全画面に ViewModel を入れるか、意図的に thin View パターンを統一するか決定すべき。

Medium Firestore リスナーコールバックのスレッド安全性 ▼

addSnapshotListener のクロージャはバックグラウンドスレッドで呼ばれる可能性があるが、@MainActor の self.towels に直接アクセスしている。Task { @MainActor in } でラップするか、Firestore のリスナーディスパッチキューを明示的に設定すべき。

Medium ステータス色定義の不整合 & 重複 ▼

TowelStatus.color プロパティは文字列 "yellow" を返すが、View 側は .orange を使用。色決定ロジックが TowelRowView, TowelDetailView, ConditionCheckDetailView, ConditionCheckRowView の 4 箇所に重複。TowelStatus に var swiftUIColor: Color を追加して統一すべき。

Medium Towel.Equatable の不完全な比較 ▼

records.count と conditionChecks.count のみで比較しているため、中身が変わっても件数が同じなら等しいと判定される。レコードの内容変更が UI に反映されない可能性がある。

Low その他 (8件) ▼
  • DateFormatter の毎回生成 (キャッシュすべき)
  • DispatchQueue.main.asyncAfter → Task.sleep に移行
  • エラー型の不統一 (AuthService が NSError を手動生成)
  • Bundle.main.icon の重複定義 (TowelApp.swift と SignInView.swift)

  • NotificationService がコールバックベース (iOS 17+ なら async/await 版 API 使用可能)

3 セキュリティ

19件の脆弱性を検出。API 認証の欠如とレート制限なしが最大のリスク。

緊急対応推奨 TOP 3

1. 状態診断 API に認証がない — URL を知っていれば誰でも Bedrock (Claude) を無制限に呼び出せる。AWS 料金の不正膨張リスク。
2. 全 Lambda にレート制限なし — API Gateway のスロットリング未設定。ブルートフォース攻撃が可能。
3. alexa-auth.html のリダイレクト先未検証 — Open Redirect によるトークン窃取のリスク。

Critical 状態診断 API (assess-nova) に認証チェックなし ▼

現状

lambda/towel-condition-assess-nova/index.mjs は Firebase Auth トークンの検証を行っていない。API URL を知る第三者が無制限に Bedrock API を呼び出せる。

リスク

AWS 料金の不正膨張 (Bedrock 呼び出し 1 回あたり $0.01-0.05)。正規ユーザーへのサービス妨害。

改善策

リクエストヘッダーに Firebase ID Token を含め、Lambda 側で admin.auth().verifyIdToken() で検証する。または API Gateway に Lambda Authorizer を追加。

Critical 全 Lambda エンドポイントにレート制限なし ▼

現状

/restore-auth, /alexa-link, /assess-nova の全エンドポイントにレート制限がない。特にリストアコード認証はブルートフォース攻撃に対して脆弱 (30^12 = 約 59 ビットのエントロピーだが、制限なしでは攻撃可能)。

改善策

API Gateway で Usage Plan + スロットリング設定 (例: IP あたり 10 リクエスト/分)。Lambda 側で連続失敗時の遅延実装。

High Firestore: inviteCodes を誰でも削除可能 ▼

allow read, create, delete: if request.auth != null; で、認証済みユーザーが他人の招待コードを削除できる (DoS)。偽コード作成やコード列挙による不正参加も可能。

改善策

招待コード CRUD を Admin SDK (サーバーサイド) に移行するか、delete をグループオーナーのみに制限。

High Storage: グループ写真が全認証ユーザーに公開 ▼

/groups/{groupId}/** が allow read, write: if request.auth != null;。グループ外のユーザーが写真を閲覧・上書き・削除可能。ファイルサイズ・コンテンツタイプの検証もなし。

改善策

最低限、write にファイルサイズ制限 (10MB) と JPEG のみ許可を追加。

High alexa-auth.html: リダイレクト先の未検証 (Open Redirect) ▼

redirect_uri パラメータを検証せずにリフレッシュトークン付きでリダイレクトしている。攻撃者が細工した URL で被害者のトークンを窃取可能。

改善策

const ALLOWED_HOSTS = ['layla.amazon.com', 'pitangui.amazon.com', 'alexa.amazon.co.jp'];
const url = new URL(params.get('redirect_uri'));
if (!ALLOWED_HOSTS.includes(url.hostname)) return;
High Alexa 連携コードのエントロピー不足 (6文字 / 30^6) ▼

6 文字 (30^6 ≈ 7.3億通り / 約 30 ビット) で有効期限 10 分。レート制限なしなら毎秒 1,214 回で 10 分以内に全探索可能。

改善策

API Gateway スロットリング設定が最優先。加えてコード長を 8 文字以上に増やす (30^8 ≈ 6,560 億通り)。

5 パフォーマンス & スケーラビリティ

15件の改善点。サブコレクション全件リスナーと Lambda のコールドスタートが主要課題。

Critical サブコレクションリスナーの無制限増殖 (1+2N 問題) ▼

現状

タオル N 個に対して 1 + 2N のリアルタイムリスナーが常時接続。タオル 10 個で 21 リスナー、50 個で 101 リスナー。全 records / conditionChecks をページネーションなしで取得。

影響

1 年間の使用 (1 タオルあたり ~120 レコード + ~24 診断) で、100 ユーザーの場合 Firestore 日次リード数が無料枠の 4.3 倍に到達。

推奨

サブコレクションは遅延ロードに変更。タオル詳細画面遷移時に初めて購読し、.limit(20) + ページネーション。リード数 90% 削減の見込み。

High firebase-admin のコールドスタート遅延 (2-5秒) ▼

3 つの Lambda が firebase-admin v12 (~56MB) に依存。コールドスタートが 2-5 秒。特に Alexa スキル Lambda は Alexa の 8 秒タイムアウトに近づく。

推奨

ES Modules の部分インポート (firebase-admin/firestore, firebase-admin/auth) で tree shaking。または Lambda Layer で共有しデプロイパッケージを軽量化。コールドスタート 0.5-1 秒に短縮の見込み。

High Alexa Lambda の N+1 クエリ ▼

fetchTowels() でタオルごとに records サブコレクションを個別クエリ。タオルドキュメントの lastExchangedAt を直接使えばサブコレクションクエリは不要。リード数 N+1 → 1 に削減。

High AsyncImage にキャッシュ戦略なし ▼

Firebase Storage の download URL にはトークンパラメータが含まれ、同じ画像でもキャッシュミスが発生しうる。診断履歴 20 件で 20 枚の並行ダウンロード。CachedAsyncImage (自作 or ライブラリ) でトークンを除いたパスベースのキャッシュキーを使用すべき。
